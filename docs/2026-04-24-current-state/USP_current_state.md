# ArtMap — USP (Unique Selling Proposition)

> **작성 시점**: 2026-04-24 **목적**: 현재 코드베이스가 실제로 제공하는 차별점을
> 객관적으로 정리. 마케팅 카피가 아닌 **구현된 기능에서 추출한 USP**.

---

## 1. 한 줄 USP

> **"내가 보고 싶은 작품이 _지금_ 어느 미술관에 있는지를 지도에서 본다."**

타 서비스(Wikipedia, Google Arts & Culture, 각 미술관 공식 사이트)와의 결정적
차이는 *원소장처*가 아닌 **현재 위치**를 별도 모델링한다는 점이다.

---

## 3. USP를 뒷받침하는 기술 결정

### 3-1. "현재 위치"를 모델링하기 위한 데이터 분리

`artworks` 테이블 한 줄에 `museum_id`를 두지 않고:

- `artwork_ownerships` — _소유_ 관계
- `artwork_locations` — _현재 위치_ 관계 (`location_type` enum)

…를 별도 테이블로 분리. 덕분에 `on_loan` 상태에서 두 미술관을 동시에 가리킬 수
있고, 향후 "전시 일정 타임라인" 같은 확장이 가능하다. Spec과 API 모두
`primary_museum` / `current_location` 두 필드를 분리 응답한다
(`/api/v1/artworks/[id]/route.ts`의 3-쿼리 패턴).

### 3-2. 지도 비용 0원 스택

- 타일: **OpenFreeMap** (API 키·과금 없음)
- 라이브러리: **MapLibre GL JS** (오픈소스 포크)
- 래퍼: **`react-map-gl/maplibre`** — `react-map-gl` 직접 import는 Mapbox 유료로
  라우팅되므로 코드 규칙으로 차단됨

→ 사용자 단위 비용이 0이므로 "비로그인 무료 공개"가 성립한다.

### 3-3. 검색 UX (PR #19)

- ARIA combobox 패턴 (스크린 리더 호환)
- 300ms debounce + `AbortController` 레이스 컨디션 방지
- 화가/작품/미술관 **3-카테고리 통합** (별도 필터 토글 없음 — "내가 뭘 찾는지
  몰라도 검색되는" UX)

### 3-4. 서버리스 친화 DB 레이어

- `@neondatabase/serverless` HTTP 드라이버 → Vercel Serverless Function의 콜드
  스타트 시 TCP 핸드셰이크 비용 회피
- Drizzle ORM (신규 코드) + `pg` Pool (레거시) 듀얼 운영 — 점진적 마이그레이션

## 4. USP의 약점·한계 (객관 기술)

USP를 약화시키는 현재 상태:

| 한계                       | 설명                                                                                                                                                          |
| -------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 데이터 깊이                | 실제 운영 데이터는 미공개. 모드 전환(`NEXT_PUBLIC_USE_MOCK=true`)이 필요한 단계로, "전 세계 작품"이라는 캐치프레이즈를 뒷받침할 데이터셋 규모가 검증되지 않음 |
| 발견성 한계                | `/artists`, `/museums` 목록 페이지가 placeholder — 검색을 모르면 화가/미술관에 도달할 수 없음                                                                 |
| 다국어 UI                  | 데이터 필드는 한/영 분리이나 UI 텍스트는 한국어 고정. 영어권 사용자에게는 USP가 전달되지 않음                                                                 |
| 작품 카드 미술관 링크 부재 | 화가 상세의 작품 카드에서 미술관으로 다이렉트 이동 불가 (debt #2) — "지도 발견" USP의 동선 일부 단절                                                          |
| 화가 메타 누락             | 작품 상세에서 화가 국적·생몰년 미표시 (debt #1)                                                                                                               |
| 응답 필드 불일치           | `museums/{id}/artworks`만 `year_label`/`thumbnail_url`, 나머지는 `year_created`/`image_url` (debt #5). API 일관성 부족                                        |
| 모니터링 없음              | 사용 패턴·실패 응답을 외부 수집하지 않아 USP 가설 검증이 어려움                                                                                               |

## 5. 사용자 가치 시나리오 (현재 구현 기준)

| 시나리오                                                                | 현재 가능 여부                                               |
| ----------------------------------------------------------------------- | ------------------------------------------------------------ |
| "고흐의 *별이 빛나는 밤*이 어디 있지?" → 검색 → 작품 상세 → 미술관 정보 | ✅ 가능                                                      |
| "파리에 가는데 모네 작품 있는 곳을 지도에서 보고 싶다"                  | ⚠️ 화가 상세의 국가 지도까지는 가능, 도시 단위 필터는 미구현 |
| "이 미술관이 가지고 있는 작품들을 보고 싶다"                            | ✅ 미술관 상세 → 작품 갤러리                                 |
| "특정 사조(인상주의)의 작품 모음을 보고 싶다"                           | ⚠️ API 필터(`movement`)는 있으나 UI는 placeholder            |
| "내가 좋아하는 작품을 저장하고 싶다"                                    | ❌ 인증·즐겨찾기 미구현                                      |
| "이 작품이 다음 달엔 어디 가 있을까?"                                   | ❌ 전시 일정 모델 없음 (`artwork_locations`은 현재 시점만)   |

## 6. 결론

ArtMap의 USP는 **"위치"라는 축으로 미술 정보를 재정렬한 것**이다. 데이터
모델·API·UI 모두에 그 의도가 일관되게 반영되어 있다 — 특히 `on_loan` 추적, 지도
줌-드릴다운, 화가의 국가 분포 시각화는 다른 미술 정보 서비스에서 보기 드문
조합이다.

다만 USP의 잠재력은 **데이터 규모와 발견성(목록 페이지) 구현**에 의해 좌우된다.
두 placeholder 페이지를 본구현으로 전환하기 전까지는 "검색을 알고 들어온
사용자"에게만 USP가 전달된다.
