# ArtMap 프로젝트 — AI 에이전트 공통 규칙

## 프로젝트 개요

- **서비스명**: ArtMap
- **설명**: 전 세계 미술 작품의 소장 위치를 지도 기반으로 탐색하는 웹 서비스
- **아키텍처**: 풀스택 Next.js — 프론트엔드(App Router)와 백엔드(API Routes)를
  하나의 Next.js 프로젝트로 처리. 별도 백엔드 서버 없음
- **배포**: Vercel 단일 배포

```
브라우저
  └── Next.js (Vercel)
        ├── App Router  → 화면 렌더링
        └── API Routes  → 데이터 처리 → PostgreSQL / PostGIS (Neon)
```

---

## 기술 스택

| 영역         | 기술                                         | 비고                                               |
| ------------ | -------------------------------------------- | -------------------------------------------------- |
| 프레임워크   | Next.js (App Router)                         | 프론트 + 백엔드 통합                               |
| 언어         | TypeScript strict 모드                       | any 타입 사용 금지                                 |
| 스타일링     | Tailwind CSS                                 | 유틸리티 클래스만 사용                             |
| 지도         | MapLibre GL JS + react-map-gl                | OpenFreeMap 타일, API 키 불필요                    |
| ORM          | Drizzle ORM (신규 코드) + `pg` Pool (레거시) | 점진적 마이그레이션 중. 신규 라우트는 Drizzle 사용 |
| DB 드라이버  | `@neondatabase/serverless`                   | Vercel 서버리스 HTTP 드라이버                      |
| 데이터베이스 | PostgreSQL (Neon 호스팅)                     | PostGIS 확장                                       |
| 마이그레이션 | `drizzle-kit`                                | Drizzle 스키마 기반                                |
| 배포         | Vercel                                       | 프론트 + API Routes 통합                           |

> **지도 import 경로 주의**: 반드시 `react-map-gl/maplibre` 사용. `react-map-gl`
> 직접 import 시 Mapbox(유료)로 연결됨.

> **DB 계층 주의**: 신규 API Route는 Drizzle ORM으로 작성한다. `pg` Pool + Raw
> SQL은 기존(레거시) 라우트에 한해 유지하며 점진적으로 Drizzle로 이관한다. DB
> 스키마의 단일 정의는 `src/lib/schema.ts`(Drizzle)이다.

## 디렉토리 구조

```
src/
├── app/          → 페이지 라우팅 + API Routes (api/v1/ 하위)
├── components/   → 재사용 컴포넌트
├── lib/          → 유틸리티, API 클라이언트, DB 연결·스키마 (db.ts, schema.ts)
├── hooks/        → 커스텀 React 훅
├── types/        → TypeScript 타입 정의
└── mocks/        → Mock 데이터
```

> DB 스키마의 단일 진실 원천은 `src/lib/schema.ts`(Drizzle 정의)이다. 별도
> `sql/` 폴더는 존재하지 않으며, 마이그레이션은 `drizzle-kit`으로 관리한다.

## 코딩 스타일

이 프로젝트의 모든 코딩 스타일 규칙은 `.gemini/styleguide.md`에 단일 진실
원천으로 정의되어 있습니다. 파일 네이밍, 타입 정의 방식, 들여쓰기, 따옴표 등
모든 스타일 결정은 styleguide를 따릅니다.

이 파일(AGENTS.md)은 도메인 규칙(다국어, UUID, 에러 응답 등)과 금지 사항만
정의합니다.

---

## 도메인 규칙

- **TypeScript**: strict 모드. `any` 타입 사용 금지 (`unknown` 또는 구체적 타입
  사용)
- **다국어**: 모든 텍스트 필드는 한/영 분리 (`_ko`, `_en` 접미사)
- **ID**: UUID 형식
- **API 에러 응답**: RFC 9457 (`application/problem+json`) 형식

---

## 금지 사항

- `any` 타입 사용
- `console.log` 커밋 (디버깅 후 반드시 제거)
- 하드코딩된 API URL (환경 변수 사용)
- CSS Modules, styled-components 등 Tailwind 외 스타일링
- `react-map-gl` 직접 import (반드시 `react-map-gl/maplibre` 경유)
- 메타 서술(사람 이름·협업·부채·향후 작업·미구현)을 파일·커밋에 작성 (→ 아래

## 파일 작성 금지 — 메타 서술 분리 원칙

**원칙**: 코드 파일과 산출 문서는 "현재 동작하는 결과물의 사실"만 담는다.
사람·협업·이력·계획·미구현 상태 등 메타 서술은 파일의 본질이 아니다. 이런 정보는
파일·커밋에 한 글자도 남기지 말고, 각 에이전트의 보고 채널에서만 전달한다. (이는
명세서 v2.0의 "계약은 파일, 배경·이력은 외부" 분리 원칙을 코드 전반에 확장한
것이다.)

### 어떤 파일에도 절대 작성하지 않는다 (코드 주석·문서·README·생성 파일 전부)

1. **개인 실명·닉네임** — 팀원 이름을 코드/문서/주석에 쓰지 않는다. 역할
   표기("백엔드 담당")조차 쓰지 말고, 가능한 한 기술 사실만 서술한다.
2. **협업·합의·논의 서술** — "~와 협의", "~와 합의 완료", "팀 논의 필요",
   "백엔드와 조율 후", "확인 받음" 류의 과정 서술 일체.
3. **기술 부채(tech debt)** — "알려진 부채", "임시 처리", "추후 리팩터 필요",
   "// TODO: 개선" 등 부채 항목·표·주석 일체.
4. **향후 작업·로드맵** — "다음 단계", "추후 구현 예정", "post-MVP", "// TODO:
   나중에", "백로그" 등 미래 계획 일체.
5. **미구현·미완 상태 서술** — "아직 미구현", "placeholder", "현재 동작 안 함",
   "임시 stub", "미완성" 등 상태 메타 서술 일체.
6. **위 내용을 담은 문서 파일의 자발적 생성** — plan.md / TODO.md / NOTES.md /
   CHANGELOG 류를 사용자가 해당 작업에서 명시적으로 요청하지 않았는데 만들지
   않는다.

### 대신 — 각 에이전트의 보고 채널에만 말한다

위 1~5에 해당하는 내용이 발생하면 파일·커밋에 쓰지 말고, 작업을 수행한
에이전트의 보고 채널에서만 서술한다.

- 코드 작성 에이전트(예: Claude Code): 작업 종료 후 **터미널 요약의 `[보고]`
  절**
- 리뷰 에이전트(예: Gemini): **PR 리뷰 코멘트**

어느 경우든 이 서술은 어떤 파일에도, 어떤 커밋 메시지에도 남기지 않는다.

예) 코드 작성 에이전트의 터미널 출력:

```
구현 완료. tsc·lint 통과.
[보고] 화가 카드의 미술관 링크는 이번 범위에서 제외했습니다.
        API 응답에 nationality 필드가 없어 후속으로 백엔드 조정이 필요합니다.
```

### 예외

사용자가 해당 작업에서 명시적으로 "이 내용을 파일에 작성하라"고 지시한 경우에
한해 허용한다(예: Plan-First 워크플로의 plan.md 작성 요청). 지시가 없으면
기본값은 항상 "파일 금지 · 보고 채널로만".

### 커밋 메시지

커밋 메시지(Conventional Commits)에도 실명·협의 서술을 넣지 않는다. 본문에는
변경의 기술적 사실만 적는다.
